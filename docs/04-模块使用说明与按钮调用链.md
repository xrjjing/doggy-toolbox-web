# 模块使用说明与按钮调用链

## 1. 这份文档怎么用

这份文档不是介绍“页面长什么样”，而是说明：

- 用户在页面点了什么
- Vue 组件里调用了哪个方法
- Store 调了哪个 `window.doggy`
- Main 里最终落到哪个 Service
- 数据最后写到哪个文件

如果你需要快速定位问题，这是最实用的一份文档。

---

## 1.1 迁移状态口径

本文后续出现的模块状态按下面三类理解：

- **已对齐**：新仓已有真实可用入口、数据链路或纯函数闭环，能作为当前主实现继续维护。
- **轻量替代**：核心能力已有，但旧项目独立页面、复杂交互、视觉细节或高级状态没有一比一迁移。
- **未完成**：旧项目能力还没有形成新仓可用入口，或仍缺独立高级 UI / 持久化 / 测试闭环。

当前大体判断：

- 已对齐：命令管理、凭证管理、Prompt 模板、节点本地资料库、HTTP 集合、备份恢复、旧数据导入、AI SDK Bridge、本机运行时检测。
- 轻量替代：工具工作台里的旧工具算法入口、12 个高级工具面板、SDK-only AI 设置页、外观系统、HTTP 格式转换、节点分享链接导入。
- 未完成：高级工具的独立页面级完整体验、窗口级标题栏动态重建、旧项目 50+ 页面逐页 Vue 化。

当前已迁移：

### 3.2 运行工具按钮

点击“运行工具”后，并不会走 Main。

它是纯前端算法：

```text
点击“运行工具”
  -> runTool()
  -> 根据 activeTool 分支调用 utils/core-*.ts
  -> 输出 result / extra
```

### 3.3 工具页 AI 分析按钮

点击“让 Codex / Claude 分析”后，会走 AI 主链路：

```text
点击“让 AI 分析”
  -> buildToolAiAssistPrompt(...)
  -> aiStore.startChat(...)
  -> AI SDK Bridge 主链路
```

也就是说：

- 工具计算本身在前端
- AI 解释结果在主进程

### 3.4 高级面板工具调用链

这类工具不走 `runTool()`，而是各自面板内部维护复杂状态，再回传最小快照：

```text
点击高级工具导航
  -> ToolWorkbenchView 切换 activeTool
  -> 渲染对应 ToolPanel 组件
  -> ToolPanel 内部执行图片/二维码/RSA/WebSocket 逻辑
  -> emit('snapshot', { input, output, extra })
  -> ToolWorkbenchView.updatePanelSnapshot(...)
  -> buildToolAiAssistPrompt(...)
  -> aiStore.startChat(...)
```

当前这条链路适用于：

- `QR Code`
- `Image Base64`
- `Mock`
- `Diff`
- `Crypto`
- `RSA`
- `WebSocket`
- `Git`
- `Docker`
- `Docker Service`
- `Docker Swarm`
- `Nginx`

### 3.5 工具页 AI 入口如何受设置页控制

```text
AiSettingsView 保存 settings
  -> aiSettingsStore.save()
  -> window.doggy.saveAiSettings()
  -> ipcMain.handle('ai:save-settings')
  -> AiSettingsService.saveSettings()

ToolWorkbenchView 点击“让 AI 分析”
  -> aiStore.startChat({ moduleId: 'tools', ... })
  -> aiSettingsStore.assertFeatureEnabled('tools')
  -> aiSettingsStore.buildMessages(...)
  -> window.doggy.aiStartChat(...)
```

也就是说：

- 工具页 AI 入口不是写死开放，而是受 `AI 设置 -> 全局开关 + tools 模块开关` 控制。
- 默认工作目录和系统提示也会通过 `aiSettingsStore` 注入到真正发送给本机 SDK 的会话中。

## 12. 一个接手者最常见的问题

### 问题：为什么很多按钮点下去后都会 `await store.xxx()` 再 `load()` 一次？

原因是当前项目采用的是“主进程为准”的资料管理方式：

- 真正权威数据在 Main 和本地文件
- Renderer 不试图本地手拼最终状态
- 保存后重新 `load()`，可以统一拿到主进程规范化后的结果

这种方式性能不是最激进，但在桌面工具项目里更稳，更适合接手维护。
